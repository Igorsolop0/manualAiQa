# MEMORY.md - API Knowledge Base

> This logic details endpoint mappings, Swagger references, and specific BackOffice/Website/Wallet flows.

## 🏛️ System Architecture: The Three Pillars

The platform backend is strictly divided into three distinct sub-systems. **Never mix their endpoints or authentication methods.**

---

### 1. Website Web API (Player-facing / "Frontend")
This handles everything a player does on the actual casino website (login, register, play, claim bonuses).
- **Swagger URL:** `https://websitewebapi.{env}.sofon.one/swagger/index.html?urls.primaryName=API+v3`
- **Format:** GraphQL (mostly for data fetching) and REST (for specific actions).
- **Authentication:** `Bearer <token>` inside the `Authorization` header.
- **Token Source:** Must be generated by registering/logging in via the UI.
- **Agent Workflow (Token Handshake):** Because I (Cipher) cannot interact with the UI, the **QA Agent** must perform a UI login via Playwright, extract the `access_token` from the GraphQL response (or LocalStorage), and save it to `~/.openclaw/workspace/shared/session/player_token.txt`. I will read this file to perform API calls as the player.

---

### 2. Backoffice Web API (Admin-facing / "Control Panel")
This handles all administrative actions, brand configuration, bonus creation, and player management.
- **Swagger URL:** `https://adminwebapi.{env}.sofon.one/swagger/index.html?urls.primaryName=API+v3`
- **Format:** Standard REST API.
- **Authentication:** Admin/Manager access token or static `UserId` header (depends on the environment configuration).
- **Use cases:** Changing player balances via Support endpoints, checking bonus `TurnoverCount`, modifying game providers.

---

### 3. Wallet Service API (Financial / "The Bank")
A high-performance microservice dedicated solely to balance operations.
- **Swagger URL:** `https://wallet.{env}.sofon.one/swagger/index.html`
- **Format:** REST API.
- **Authentication:** **None (Open API/Internal)**. Requires only `{partnerId}` and `{clientId}` in the URL paths.
- **Use cases:** Fast balance checks, Add/Remove money (`transaction/correction/credit` vs `transaction/correction/debit`).
- **Critical Rule:** When you need to quickly check a balance or add money for a test, **always prefer the Wallet Service** over the Backoffice API because it's simpler and doesn't require session tokens.

---

## 🔒 Authentication Handshakes & Cross-Agent Data Flow

If I am asked to test an API mutation that requires a Player Session, I cannot do it alone. The flow must be:
1. `Nexus` commands `QA Agent` to register/login a player via Web UI.
2. `QA Agent` performs the action and writes the exact `Authorization: Bearer <token>` to `~/.openclaw/workspace/shared/session/player_token.txt`.
3. `QA Agent` tells `Nexus` that the token is ready.
4. `Nexus` commands me (`API Agent`) to execute the API mutation.
5. I read the token from the shared folder and execute the `curl` or `fetch` script.
